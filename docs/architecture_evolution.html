🔴 구조 결함 1: 지식이 프롬프트에만 존재

Cross-device 우선순위, CE/CT 구분, 카테고리 별칭이 자연어 프롬프트 안에만 있다. LLM은 확률적이어서 같은 쿼리에서도 결과가 달라진다.

"Primus > Solar8000 for ventilator" (프롬프트)
_CE = effect-site (하드코딩 가정)
vasopressor → Medication (수동 매핑)

🟡 구조 결함 2: 문자열 키워드 매칭

ILIKE '%keyword%' 검색은 동의어, 하이픈, 약어에서 실패한다. 정보 손실이 3단계(키워드→DB→LLM)에 걸쳐 누적된다.

"effect site" ≠ "Effect-site" → 0건
"amount" ≠ "Volume" → 0건
"consciousness" ≠ "BIS" → 0건

🟣 구조 결함 3: 파이프라인이 항상 끝까지 실행

Ambiguity를 감지해도 멈추지 않는다. auto_resolve_ambiguity는 설정만 있고 소비 코드가 없다. caseid 없이도 숫자를 반환한다.

"first 10 min HR?" → 76.01 반환 (FAIL)
ambiguities[] 있어도 다음 단계 진행
AnalysisAgent가 첫 케이스를 임의 선택
↓ 세 가지 결함의 공통 원인 ↓

"파라미터에 대한 의미 지식이 인덱싱 시점에 계산되지 않고, 런타임에 LLM 규칙으로 추론된다"

IndexingAgent가 param_key, semantic_name, unit만 저장하고, 나머지 의미 정보(측정 타입, 장비 간 관계, 개념 동일성)를 저장하지 않는다.
그 결과 ExtractionAgent가 런타임마다 규칙+LLM으로 이를 추론하려 하고, 이 추론이 VitalDB-specific 하드코딩에 의존하게 된다.

Part B Phase1
Semantic Index — IndexingAgent가 저장해야 할 것

parameter 테이블 확장 스키마

이 6개 필드가 추가되면 ExtractionAgent의 모든 VitalDB-specific 규칙을 데이터 기반으로 대체 가능

embedding vector(1536)
semantic_name + unit + description의 텍스트 임베딩. 의미 검색의 기반. 인덱싱 시 1회 계산.
text-embedding-3-small, 259개 = $0.001 1회성
measurement_type varchar
unit 패턴으로 자동 추론. 어떤 데이터셋이든 단위는 국제표준(SI)이므로 범용 적용 가능.
mL/hr→rate | mL→cumulative | Hz→waveform | mcg/mL→concentration
concept_id varchar
동일 개념의 파라미터를 하나로 묶는 ID. 임베딩 클러스터링으로 자동 생성. VitalDB 이름 불필요.
"TIDAL_VOLUME": [Solar8000/VENT_TV, Primus/TV]
concept_priority int
같은 concept_id 내에서의 우선순위. 프롬프트 규칙이 아닌 DB 필드로 결정론적 선택.
Solar8000/VENT_TV:1 > Primus/TV:2 > Primus/SET_TV:3
device_group varchar
file_group 메타데이터에서 추론. VitalDB 이름 아닌 역할 기반 (patient_monitor, ventilator).
patient_monitor | anesthesia_machine | infusion_pump
variant_type varchar
같은 개념의 변형 구분. SET_/VENT_ 같은 prefix 패턴이 아닌 의미 기반.
measured | target_setting | derived | waveform

T-02
Ontology/Category		 "vasopressor" → keyword 검색 0건. Neo4j 수동 매핑 필요 Embedding 검색: "vasopressor" → cosine similarity → NEPI_RATE, DOPA_RATE 등 자동 매칭 ParameterEmbeddingCache, concept_id 클러스터 범용
T-03
Cross-Device		 "Primus 우선" 규칙이 프롬프트에 하드코딩. LLM이 확률적으로 적용 concept_id 그룹 내 concept_priority 필드로 결정론적 선택. 프롬프트 규칙 불필요 concept_priority DB 필드, Semantic Index 범용
T-05
clarify↔retrieve 역전		 후보 없음 = Unknown 카테고리 → clarify. 복합 조건 쿼리도 clarify로 오분류 Embedding으로 항상 의미적 후보 확보 → LLM이 더 좋은 후보로 정확히 판단. not_found vs clarify 기준 명확화 Hybrid search, scope_intent 범용
T-08
Waveform vs Scalar		 _WAV suffix 의존 → VitalDB 전용. 타 데이터셋 불가 measurement_type='waveform' DB 필드로 결정론적 필터. unit(Hz, µV) 패턴으로 자동 추론 measurement_type Semantic Index 필드 범용
T-09
Volume vs Rate		 _VOL/_RATE suffix 의존 → VitalDB/Orchestra 전용 measurement_type='cumulative' vs 'rate' DB 필드. unit: mL→cumulative, mL/hr→rate (SI 표준) measurement_type Semantic Index 필드 범용

Part B 목표

핵심 원리: unit은 국제 표준(SI)이라 어느 데이터셋이든 mL/hr은 rate, mL은 cumulative다. semantic_name의 의미는 의학 도메인에서 공통이라 embedding 유사도가 데이터셋 무관하게 작동한다. concept_id 클러스터링은 비지도 학습이라 레이블이 필요 없다. 결국 IndexingAgent가 새 데이터셋에 Semantic Index를 구축하면, ExtractionAgent는 아무 코드 변경 없이 작동한다.
